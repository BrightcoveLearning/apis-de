    Implementieren des Wiedergaberechten

    In diesem Thema erfahren Sie, wie Sie die Videowiedergabe mit dem Playback Rights Management Service von Brightcove verwalten.
    Diese Funktion steht für eine bestimmte Gruppe von Kunden mit Zugriff auf die Phase der begrenzten Verfügbarkeit des Playback Rights Management Service zur Verfügung.

    Einführung

    Der Playback Rights Management Service von Brightcove bietet eine skalierbare und ausdrucksstarke Methode zur Verwaltung der Videowiedergabe.

    Wenn Sie mit dieser Funktion nicht vertraut sind, lesen Sie die Übersicht: Dokument „Wiedergaberechte verwalten “

    Validierungsprozess

    Wiedergaberechte werden in der Reihenfolge ihrer Spezifität und Übereinstimmung angewendet. Eine Zulassungsregel negiert den Rest der Regeln, da sie weniger spezifisch sind als diejenige, die die Regel zulässt.

    Möglicherweise möchten Sie eine bestimmte IP zulassen, um eine Länderregel für diese IP zu vermeiden. Möglicherweise möchten Sie auch eine andere IP-Adresse blockieren, die normalerweise durch die Länderbeschränkung zulässig ist. Es kann also sinnvoll sein, beides zu haben block-ips und allow-ips in der gleichen Definition der Wiedergaberechte. Gleiches gilt für andere Regeln.

    Sie können Regeln für die meisten Rechte zulassen und blockieren. Länder sind die einzigen, in denen es möglicherweise nicht sinnvoll ist, beide zu haben.

    Die folgenden Flussdiagramme zeigen, wie der Validierungsprozess funktioniert.

    Workflow

    Gehen Sie folgendermaßen vor, um die Wiedergabebeschränkungen zu verwalten:

    1. Richten Sie Ihr Konto ein
    2. Einschränkungen definieren
    3. Verknüpfen Sie Einschränkungen mit einem Video

    4. Optional: Definieren Sie Einschränkungen auf Benutzerebene

    5. Konfigurieren Sie Ihren Player

    Einschränkungen definieren

    Verwenden Sie die API für Wiedergaberechte, um Einschränkungen für die Videowiedergabe zu definieren.

    API für Wiedergaberechte

    Jedes Objekt mit Einschränkung der Wiedergaberechte kann mit einem oder mehreren Videos verwendet werden.

    Basis-URL

    Die Basis-URL für die API lautet: 

    https://playback-rights.api.brightcove.com/v1

    Konto-Pfad

    In allen Fällen werden Anfragen für ein bestimmtes Video Cloud-Konto gestellt. Daher fügen Sie immer den Begriff Konten gefolgt von Ihrer Konto-ID zur Basis-URL hinzu: 

    https://playback-rights.api.brightcove.com/v1/accounts/{accountID}

    Autorisierung

    Ein Zugriffstoken für Anfragen ist erforderlich und muss im Authorization-Header vorhanden sein: 

    Authorization: Bearer {access_token}

    Das Zugriffstoken ist ein temporäres OAuth2-Zugriffstoken, das vom Brightcove OAuth Service bezogen werden muss. Weitere Informationen zum Abrufen von Client-Anmeldeinformationen und zum Abrufen von Zugriffstoken finden Sie in der Brightcove OAuth Overview.

    Berechtigungen

    Anfragen an die Playback Rights API müssen von Client-Anmeldeinformationen mit den folgenden Berechtigungen gestellt werden:

    • video-cloud/playback-auth/playback-rights/read
    • video-cloud/playback-auth/playback-rights/write

    Beschränkungen verwalten

    Die Wiedergaberechte-API unterstützt die folgenden Anforderungen. API-Details finden Sie in der API-Referenz für Wiedergaberechte.

    Erstellen Sie neue Wiedergaberechte

    POST /v1/accounts/{accountID}/playback_rights
  Content-Type: application/json
  Body: {playback rights object}

    Eine Liste der gültigen Felder finden Sie im Abschnitt Request body .

    Curl-Beispiel

    Erstellen Sie ein Wiedergaberecht, das nur für Australien gültig ist. 

    curl -X POST
  https://playback-rights.api.brightcove.com/v1/accounts/{your_account_id}/playback_rights \\
  -H 'Autorisierung: Inhaber {access_token} '\\
  -H 'Content-Type: Anwendung/JSON'\
  -d '{
  "geo": {
    "allow_countries": [
      "AU"
    ]
  }
} '

    API Antwort

    Speichern Sie die playback_rights_id Wert für spätere Verwendung. Sie finden diesen Wert in der API-Antwort. Entweder:

    • Antwortheader::

      Das playback_rights_id Wert finden Sie in der Location Feld der Header-Antwort. Es sollte ähnlich sein: 

      Location: /v1/accounts/your account_id/playback_rights/your playback_rights_id

    • Antwortkörper

      Das playback_rights_id Der Wert kann im Antworttext als der gefunden werden id Feld. Es sollte ähnlich sein: 

      {
  "id": "your playback_rights_id",
  "geo": {
  "allowed_countries": [
    "MX",
    "US"]
}

    Holen Sie sich alle Wiedergaberechte für ein Konto

    GET /v1/accounts/{accountID}/playback_rights

    Holen Sie sich ein bestimmtes Wiedergaberecht für ein Konto

    GET /v1/accounts/{accountID}/playback_rights/{playbackRightsID}

    Aktualisieren Sie ein bestimmtes Wiedergaberecht:

    PUT /v1/accounts/{accountID}/playback_rights/{playbackRightsID}
  Content-Type: application/json
  Body: {playback rights object}

    Eine Liste der gültigen Felder finden Sie im Abschnitt Request body .

    Löschen Sie eine bestimmte Wiedergabe rechts:

    DELETE /v1/accounts/{accountID}/playback_rights/{playbackRightsID}

    Hier sind die Felddetails:

    Feld Art Beschreibung
    allowed_countries, blocked_countries String Array von Ländercodes mit zwei Buchstaben, die dem ISO-3166-1-Alpha-2-Standard entsprechen. Eine Liste der Werte finden Sie in der Offiziell zugewiesene Codeelemente.
    allowed_zip_codes, blocked_zip_codes String Array von Postleitzahlen, denen das aus zwei Buchstaben bestehende Land und der Bindestrich vorangestellt sind. z.B ["US-90045"].
    Der aus zwei Buchstaben bestehende Ländercode muss in Großbuchstaben geschrieben sein und dem Alpha-2-Standard ISO 3166-1 entsprechen, wie in der Offiziell zugewiesene Codeelemente.
    allowed_dmas, blocked_dmas Ganzzahl Array von Nielsen Designated Market Area (DMA) -Nummern. Eine Liste der Werte finden Sie in der DMA-Codes Dokument.
    blocked_proxies: anonymous Boolesch Die IP-Adresse des Clients ist nicht verfügbar. Enthält Dienste, die den Standort ändern, um DRM, TOR-Punkte, temporäre Proxys und andere Maskierungsdienste zu übertreffen.
    blocked_proxies: public Boolesch Mehrere Benutzer werden von einem Standort aus vertreten, der den öffentlichen Internetzugang ermöglicht.
    blocked_proxies: corporate Boolesch Im Allgemeinen als harmlos angesehen, aber die Lage kann ein Problem sein. Identifizieren Sie, ob mehrere Benutzer über einen oder mehrere zentrale Standorte vertreten sind und eine einzige IP-Adresse für das Netzwerk gemeinsam nutzen können.
    blocked_proxies: transparent Boolesch Die IP-Adresse des Clients ist über HTTP-Header verfügbar, obwohl der Wert nicht unbedingt zuverlässig ist (z. B. kann er gefälscht werden).
    allowed_domains, blocked_domains String Array von Domainnamen
    start_time, end_time Ganzzahl Epochenzeit
    allowed_ips, blocked_ips Ganzzahl Array von IPv4 / IPv6-Adressen oder CIDR-Blöcken.
    allowed_days String Array von 3-Buchstaben-Kleinbuchstaben für die Tage, an denen die Ressource abgerufen werden darf. Eine oder mehrere von: mon, tue, wed, thu, fri, sat, sun
    allowed_hours Ganzzahl Array von Stunden ab der 24-Stunden-Uhr (beginnend bei 0 bis 47), in denen die Ressource abgerufen werden darf. 0 bis 23 für den aktuellen Tag und 24 bis 47 für das Enddatum des nächsten Tages. Wenn ein zulässiger Stundenblock am vorherigen Tag beginnt und am folgenden Tag endet, ist eine Notation von 24+ erforderlich.

    Beispiel: der Wert von 3-4 In diesem Header bedeutet dies, dass die Ressource von 3:00 Uhr UTC bis 3:59 Uhr UTC verfügbar ist
    allow_insecure Boolesch Standard: false
    Stellen Sie dies auf ein true macht das JWT-Token optional.
    disabled Boolesch Standard: false
    Stellen Sie dies auf ein true Deaktiviert die Wiedergaberechte und ermöglicht die Wiedergabe für alle.
    name String Optionaler Name für die Wiedergabe

    Verknüpfen Sie Einschränkungen mit einem Video

    Verwenden Sie die CMS API um eine Wiedergaberechten-ID mit einem Video zu verknüpfen. Sie verwenden die ID für Wiedergaberechte, die Sie im vorherigen Schritt erstellt haben.

    CMS-API

    Jedes Objekt mit Einschränkung der Wiedergaberechte kann mit einem oder mehreren Videos verwendet werden.

    Basis-URL

    Die Basis-URL für die API lautet: 

    https://cms.api.brightcove.com/v1

    Konto-Pfad

    In allen Fällen werden Anfragen für ein bestimmtes Video Cloud-Konto gestellt. Daher fügen Sie immer den Begriff Konten gefolgt von Ihrer Konto-ID zur Basis-URL hinzu: 

    https://cms.api.brightcove.com/v1/accounts/{accountId}

    Autorisierung

    Ein Zugriffstoken für Anfragen ist erforderlich und muss im Authorization-Header vorhanden sein: 

    Authorization: Bearer {access_token}

    Das Zugriffstoken ist ein temporäres OAuth2-Zugriffstoken, das vom Brightcove OAuth Service bezogen werden muss. Weitere Informationen zum Abrufen von Client-Anmeldeinformationen und zum Abrufen von Zugriffstoken finden Sie in der Brightcove OAuth Overview.

    Berechtigungen

    Anfragen an die Playback Rights API müssen von Client-Anmeldeinformationen mit den folgenden Berechtigungen gestellt werden:

    • video-cloud/video/read
    • video-cloud/video/update

    Beschränkungen verwalten

    Das CMS API unterstützt die vielen Anfragetypen. Verwenden Sie Folgendes, um ein Video zu aktualisieren:

    Ein Video aktualisieren:

    Associate a playback_rights_id mit einem Video. Diese ID sollte in der Wiedergaberechte-API vorhanden sein, die Sie im vorherigen Schritt erstellt haben. 

    PATCH /v1/accounts/{account_id}/videos/{video_id}
  Content-Type: application/json
  Body: {video object}
    Curl-Beispiel

    Fügen Sie einem Video eine playback_rights_id hinzu. 

    curl -X PATCH \\
  https://cms.api.brightcove.com/v1/accounts/Ihre account_id videos /deine video_id \\.
  -H 'Autorisierung: Träger Ihr access_token '\\
  -H 'Content-Type: Anwendung/JSON'\
  -d '{
	"playback_rights_id": "Ihre playback_rights_id""
} '

    Holen Sie sich ein bestimmtes Video:

    GET /v1/accounts/{account_id}/videos/{video_ids}

    Ausführliche Informationen zur Verwendung der API finden Sie in der CMS-API-Referenz.

    Definieren Sie Einschränkungen auf Benutzerebene

    Wenn Sie Einschränkungen auf Benutzerebene wünschen, benötigen Sie eine JSON Web Token (JWT) konfiguriert mit Ansprüchen, die mit Ihren Einschränkungen verbunden sind.

    Die Schritte

    So erstellen Sie eine JSON Web Token (JWT) Gehen Sie folgendermaßen vor, um an Ihren Spieler weiterzugeben:

    1. Einschränkungen festlegen

      Bestimmen Sie die Art der Einschränkungen auf Benutzerebene, die Sie verwenden möchten, aus der folgenden Liste. Schließen Sie die erforderlichen JWT-Ansprüche ein, wenn Sie Ihre erstellen JSON Web Token (JWT).

      Grenzwerte für Geräte

      Wenn eine DRM-Lizenzanforderung gestellt wird, wird das Gerät des Benutzers registriert und eine eindeutige ID zugewiesen. Das Gerätelimit wird bei jeder Lizenzanfrage überprüft und durchgesetzt. Um diese Einschränkung zu verwenden, generieren Sie eine JWT mit der aktuellen Benutzer-ID (UID).

      Einzelheiten finden Sie in der Implementieren von Gerätegrenzen Dokument.

      Limits für gleichzeitige Streams

      Wenn eine DRM-Lizenzanforderung gestellt wird, wird das Gerät des Benutzers registriert und eine eindeutige ID zugewiesen. Bei jeder Lizenzanforderung wird die Geräte-ID überprüft, um sicherzustellen, dass nur eine bestimmte Anzahl von Videostreams gleichzeitig angezeigt wird.

      Einzelheiten finden Sie in der Begrenzen gleichzeitiger Streams pro Viewer Dokument.

      Tags

      Basierend auf tags behaupten, die JSON Web Token (JWT) gibt die Gruppe von Videos an, die der Benutzer sehen darf. Sie können Videos mit Tags mithilfe von gruppieren Medienmodul in Brightcoves Video Cloud Studio.

      Der Wiedergabeautorisierungsdienst überprüft die mit jedem Video verknüpften Tags. Wenn mindestens eines der Tags mit der Liste im JWT-Token übereinstimmt, kann das Video angezeigt werden.

      Tags im JWT-Token werden als Array von Tags aufgelistet.

      Sie sind auf ein Array von 40 Tags im JWT-Token beschränkt.

    2. Erstellen Sie ein JSON Web Token

      Informationen zum Erstellen eines Token, das bei der Kommunikation mit der Playback-API von Brightcove verwendet werden soll, finden Sie im Dokument Creating a JSON Web Token (JWT) .
