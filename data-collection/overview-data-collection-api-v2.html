    Übersicht: Datenerfassung API v2

    In diesem Thema erhalten Sie einen Überblick über die Analytics Data Collection API v2, mit der Sie Ihren Video Cloud Analytics-Daten Ereignisse hinzufügen können, in denen Brightcove die Ereignisse nicht direkt verfolgen kann.
    Notiz: Sie sollten diese API NICHT verwenden, um Ereignisse für den Brightcove Player oder für die Android- und iOS-SDK-Player zu verfolgen, da dies bereits automatisch geschieht und Sie doppelte Daten hinzufügen würden.

    Einführung

    Analytics-Daten werden automatisch von den Brightcove Players gesendet, einschließlich der von den Native Player-SDKs bereitgestellten. Wenn Sie keinen Brightcove Player zur Bereitstellung von Video Cloud-Videos verwenden, müssen Sie den Player, den Sie verwenden, instrumentieren, um die Daten an den Data Collector zu senden.

    Datenerfassung API v2 ist der aktuelle Standard. Die Version v1 ist veraltet. Wenn Sie eine v1-Implementierung haben, lesen Sie den Abschnitt Änderungen von v1 unten.

    Neben dieser Übersicht und der API-Referenz finden Sie auch diese Beispielimplementierung.

    Die Analytics-Datenerfassungs-API ist der Endpunkt für Echtzeitanalyse-Ereignisse. Ereignisdaten werden über eine Reihe von Parametern, die über HTTP-Anfragen übermittelt werden, an Brightcove gesendet, z. B.:

      http://metrics.brightcove.com/v2/tracker?event=video_view&amp;domain=videocloud&account=123&video=789

    Diese Parameter beschreiben einen Fakt über den Zustand des Systems, als ein Ereignis eingetreten ist. Das obige Beispiel beschreibt die Tatsache, dass ein video_view Ereignis für Video 789 für das Konto 123 (oder: Ein Benutzer hat begonnen, das Konto zu sehen 123s Video 789. Nachstehend finden Sie eine Beschreibung der aktuellen Analytics-Ereignisse, die nachverfolgt werden).

    Abmessungen:

    Dimensionen sind qualitative Fakten über den Zustand des Systems, wenn ein Ereignis eintritt. Zum Beispiel, wenn die Anfrage lautet:

      http://metrics.brightcove.com/tracker
  ?event=video_view&session=581136_2018-07-03T18:34:46.214Z
  &domain=videocloud&account=123
  &video=789

    Die Video-ID ( 789) und die Konto-ID ( 123) sowie alle Geräte- und Standortinformationen, die aus der Anfrage selbst stammen, beziehen sich auf das video_view Ereignis. Das Analysesystem zeichnet auf, dass ein video_view Ereignis eingetreten ist, als diese Anfrage gestellt wurde, mit diesen Dimensionen.

    Ereignis- und Domain-

    Der event Parameter beschreibt, welches Ereignis eingetreten ist. Der domain Parameter bietet einen Namespace für Ereignisse. Das event , domain , und session sind erforderliche Parameter (der Wert von domain ist immer videocloud).

    Weitere Parameter

    Bestimmte Parameter müssen in Ereignisse einbezogen werden, damit das Analytics-System sie erfolgreich analysieren kann

    Du solltest nicht senden player (Spieler ID) oder player_name Informationen für Nicht-Brightcove-Spieler. Verwenden Sie stattdessen den destination Parameter, um zu verfolgen, wo das Video angesehen wurde.

    Arten von Reaktionen

    Die Antwort auf eine API-Anfrage zur Analytics-Datenerfassung umfasst einen HTTP-Antwortcode und eine lesbare Nachricht.

    HTTP-Statuscode Beschreibung Beispiel
    200 Die Anfrage wurde erfolgreich vom Sammler empfangen und wurde beibehalten. (gibt ein transparentes 1x1 Pixel großes GIF-Bild zurück)
    400 In der vom Client gesendeten Anfrage fehlt ein erforderlicher Parameter: domain , account oder event. (Dieser Status wird nicht zurückgegeben, wenn domänenspezifische Parameter fehlen.) "Invalid 'event' parameter"
    50x Dies ist ein Fehlercode weist auf ein Problem auf der Serverseite hin. Ihre Veranstaltung wurde möglicherweise vom Analysesystem erfolgreich aufgezeichnet. "Server-side failure, please retry."

    Minimale Daten

    Sie sollten mindestens eine session ID und ein video_view Ereignis für jedes während einer Sitzung wiedergegebene Video senden. Der video_view sollte nach Abschluss aller Pre-Roll-Anzeigen gesendet werden.

    session

    Dies ist der Sitzungsbezeichner. Das session ist im Wesentlichen eine Ansicht einer Seiten- oder App-Ansicht, in der ein Spieler enthalten ist, solange dies andauert. Der Wert sollte für die Dauer der Sitzung konstant sein und für alle Ereignisse gesendet werden. Es sollte so nah wie möglich an einem global eindeutigen Bezeichner (GUID) liegen. Wenn es zu Kollisionen kommt, können die beiden Sitzungen als ungültig verworfen werden, wenn sie nicht entwirrt werden können.

    Es gibt verschiedene Schemata zum Erstellen von GUIDs in JavaScript. Ein Beispiel ist in diesem GitHub-Repository. Bitte beachte, dass Skripte von Drittanbietern von Brightcove nicht unterstützt werden.

    Beachten Sie, dass das Analytics-System nur das zurückgeben kann, was Sie ihm geben - nicht das Senden vollständiger Daten führt zu einem Mangel an Daten in Video Cloud Analytics.

    Best-Practices

    Um sicherzustellen, dass Sie die richtigen Daten an den Collector senden, sollten Sie Ihr Datensammlungsskript testen, bevor Sie es im Allgemeinen bereitstellen. Wir empfehlen:

    1. Erstellen Sie das Datenerfassungsskript für Ihren Player.
    2. Testen Sie mindestens einen Tag in einer kontrollierten Umgebung.
    3. Überprüfen Sie die Analysedaten entweder über das Analytics-Modul oder das, Analytics API um sicherzustellen, dass das Erfasste Ihren Erwartungen entspricht.

    Senden der Anfrage - Vermeidung von CORS-Problemen

    Beachten Sie, dass der beste Weg, um an die Datenerfassungs-API zu „senden“, darin besteht, die Anforderungs-URL als src Wert für ein neues image ( img ) -Element festzulegen und dieses Element dann in die Seite zu injizieren.

    Sie sollten dieselbe Methode anstelle XMLHttpRequest einer anderen verwenden, um CORS-Probleme zu vermeiden.

    Eine Beispielimplementierung finden Sie im Beispiel zur Datenerfassung.

    Junk-Daten

    Im Allgemeinen werden Daten, die an den Collector gesendet werden, vom Analytics-System als Wahrheit aufgezeichnet. Wenn ein Ereignis unangemessene oder falsche Informationen enthält, wird das Analytics-System die Daten falsch interpretieren.

    Wenn Sie beispielsweise versehentlich den Zeitstempel als Video-ID senden, werden Ihre Analysedaten auf eine Weise verzerrt, die sich auf die Gesamtzusammenfassung auswirkt.

    Untersuchen Sie die URLs, die Sie generieren, sorgfältig, bevor Sie mit dem Senden von Daten beginnen!

    URI-Codierung

    Alle Zeichenfolgen, die Sie an die Data Collection-API senden und Leerzeichen oder Sonderzeichen enthalten können, müssen URI-codiert sein, damit die Anfrage erfolgreich ist. Wenn Sie die Anfrage über JavaScript einreichen, können Sie die encodeURI() Methode verwenden, mit der die Anforderungszeichenfolge kodiert wird. Beispiel:

      urlStr += "&video=" + currentVideo.id + "&video_name=" + encodeURI(currentVideo.video_name);

    play_request
    Absichtig/Bedeutung

    Wird gesendet, wenn die Wiedergabe entweder durch ausdrücklichen Klick des Benutzers auf die Wiedergabetaste oder automatisch initiiert wird, wenn die Plattform die Wiedergabe in einem Auto-Play-Szenario auslöst.     Eindeutiger Benutzer

    Sie können die Plugin-Funktionalität von Brightcove Player verwenden, um den gemeldeten Analysen eindeutige Video-Viewer-Daten hinzuzufügen. Dazu fügen Sie dem settings Objekt der Analytics-Funktionalität eine eindeutige Kennung hinzu.

    Wie eine eindeutige Benutzer-ID erfasst wird, ist natürlich von Anwendung zu Anwendung unterschiedlich, aber zum Beispiel geht dieser Code davon aus, dass eine Anmelde-URL erfasst wird, die eindeutige Benutzerdaten wie http://exampledomain.com/users/912389123enthält. Diese eindeutige URL wird an das Plugin übergeben.

    Der folgende Code des Plugins führt die folgenden Aufgaben aus:

    • Verwendet die Standardsyntax, um ein Brightcove Player-Plugin mit dem Namen des Plugins als zu erstellen uniqueUserForAnalyticsPlugin. Das Plugin akzeptiert auch ein options Objekt, das Daten enthält, die an das Plugin weitergegeben werden.
    • Der myPlayer Variable wird eine Referenz auf den Spieler zugewiesen. Ebenso werden zwei weitere Variablen erstellt.
    • Der userPath Variablen wird der Pfad zugewiesen, der über das options Objekt an das Plugin übergeben wird.
    • Der uniqueViewer Variable wird die geparste Version des zugewiesen userPath , sodass der Variablen nur die Benutzer-ID-Ziffern zugewiesen werden.
    • Dem settings Objekt des Analytics-Plugins wird eine Benutzereigenschaft hinzugefügt.
      videojs.registerPlugin('uniqueUserForAnalyticsPlugin', function(options) {
  var myPlayer = this,
  userPath = '',
  uniqueViewer = '';
  //Assign uniqueViewer a value according to your app and business rules
  //In this example, parsing the path passed to the plugin in the options object
  userPath = options.path;
  uniqueViewer = userPath.substring( userPath.lastIndexOf('/') + 1 );
  //Assign a user variable to Analytic's settings object
  myPlayer.bcAnalytics.client.user(USER) = uniqueViewer;
  });

    Dieser Code muss geändert werden, um Ihrer Anwendungslogik zu entsprechen, und dann in einer für das Internet zugänglichen URL gespeichert werden.

    Anstelle des folgenden JSON würden Sie die Zeichenfolge mit Benutzerdaten an das Plugin übergeben. Natürlich müsste der Plugin-Code entsprechend aktualisiert werden, um die eindeutige Benutzer-ID zu extrahieren.

      {
  "path": "http://exampledomain.com/users/912389123"

    device_type, device_os, device_os_version, device_manufacturer, und browser_type Parameter

    Standardmäßig versucht das Analytics-System, Gerätetyp- und Betriebssysteminformationen aus dem User-Agent-Header zu erkennen. Wenn beide device_type und device_os gesendet werden, werden die Informationen aus dem User-Agent-Header zugunsten von ignoriert device_type und device_os. In den meisten Fällen müssen Sie keine Geräte-, Betriebssystem- und Browserinformationen senden — diese Überschreibung sollte nur verwendet werden, wenn der User-Agent unzuverlässig oder anderweitig nicht verfügbar ist.

    Das Analytics-System zeichnet auf, other wenn eine Anfrage nicht erkannte Werte für Geräteparameter-Überschreibungen enthält.

    Geografische Daten Parameter

    Standardmäßig wird das Analytics-System versuchen, geografische Informationen von der Remote-IP-Adresse zu erkennen. Dieses Verhalten kann überschrieben werden country, indem man country_name, region, region_name, city und dma Parameter. In den meisten Fällen sind diese Parameter nicht erforderlich — diese Überschreibung sollte nur verwendet werden, wenn die Remote-IP-Adresse unzuverlässig oder anderweitig nicht verfügbar ist.

    Das Analysesystem zeichnet auf ZZ oder unknown wenn eine Anfrage unerkannte Werte für Überschreibungen enthält.

    Ziel- und Quellenparameter

    Das destination und source Parameter geben den URI an, von dem das Ereignis stammt ( destination ) und die URI, die den Benutzer dorthin geschickt hat ( source).

    Der source Parameter wird verwendet, um Informationen zur Verkehrsquelle zu bestimmen. Wenn source nicht angegeben wird, behandelt das Analytics-System Ereignisse als durch direkten Datenverkehr initiiert.

    Der destination Parameter wird verwendet, um Informationen zum Verkehrsziel zu bestimmen, d. h. wo das Video angesehen wird. Wenn der URI keine Autorität enthält, zeichnet die API keine a auf destination_domain. Der destination_path wird als Pfad im URI aufgezeichnet.

    Während der Webwiedergabe ist die URL in der Adressleiste der Seite, auf der das Video abgespielt wird destination, die und die source ist der Referrer ( top.document.referrer).

    Wenn Sie beispielsweise auf der Brightcove Support-Website nach „Live-Streaming-Wirecast“ suchen und sich ein Video ansehen, das in den Ergebnissen angezeigt wird:

    Parameter Wert
    source 
      https://support.brightcove.com/en/video-cloud/search/live%20streaming%20wirecast
    destination 
      https://support.brightcove.com/en/video-cloud/training-videos/live-streaming-wirecast

    Wenn es keine URL gibt (wie zum Beispiel bei der nativen Wiedergabe), beides destination und source sollten gültige URIs sein, die angeben, wo das Video wiedergegeben wird und wie der Benutzer dorthin gelangt ist.

    Angenommen, das destination ist ein gültiger URI:

      <scheme name> : <hierarchical part> [ ? <query> ] [ # <fragment> ]
  ex. https://www.example.com/foo/bar/baz
  --------------/----------/
  |             |
  authority        path
  ---/    -------------------------/
  |                |
  scheme       hierarchical part

    Das Analytics-System wird es wie folgt behandeln:

    Wenn der URI eine Autorität enthält, verwendet die API-Antwort diese Berechtigung als destination_domain und jeden Pfad, der als bereitgestellt wird destination_path. Wenn der URI keine Autorität enthält, zeichnet die API keine a auf destination_domain. Der destination_path wird als Pfad im URI aufgezeichnet. A destination ohne einen hierarchischen Teil (z. B. nur ein Schema) gilt als ungültig, ebenso wie jeder Wert ohne Schema.

    video_engagement: Enthält zusätzliche Wiedergabe-Auswahl, Bitratenmessungen und Pufferinformationen. Eine subtile Änderung der Videointeraktion wurde auch dadurch vorgenommen, dass sie regelmäßig gesendet werden sollte, auch wenn während der Interaktionszeit keine Anzeige stattgefunden hat. Diese Änderung soll Verzögerungen bei der Nachverfolgung von erneuten Pufferung ermöglichen, die dazu führen, dass Benutzer auf die Wiedergabe warten.

    • reichweite: Der Bereichsparameter ist jetzt optional, der Bereich kann aus einem Engagement-Ereignis ausgelassen werden, um zu zeigen, dass während des von der Veranstaltung abgedeckten Zeitraums keine Anzeigeaktivität stattfand. (zum Beispiel, wenn es nur eine Re-Puffer-Aktivität gibt)
    • rendition_url: Die URL zur zuletzt ausgewählten Wiedergabeversion. Für einen HLS-Stream wäre dies beispielsweise die URL zur zuletzt ausgewählten Variante.
    • rendition_indicated_bps: Die angegebene Bitrate der zuletzt ausgewählten Wiedergabeversion in Bits pro Sekunde.
    • rendition_mime_typ: Der Mime-Typ der zuletzt ausgewählten Formatvariante.
    • rendition_height: Die codierte Höhe der Videowiedergabe in Pixeln
    • rendition_width: Die codierte Breite der Videowiedergabe in Pixeln
    • rebuffering_sekunden: Die Anzahl der Sekunden, die der Benutzer aufgrund der nicht angeforderten Verzögerung während des Interaktionszeitraums auf die Wiedergabe des Videos gewartet hat.
    • rebuffering_count: Die Häufigkeit, mit der die Wiedergabe aufgrund der erneuten Pufferung während des dargestellten Interaktionszeitraums gestopp
    • forward_buffer_sekunden: Die Anzahl der Sekunden des Videos, die sich derzeit im Vorwärtspuffer befinden.
    • measured_bps: Das Verhältnis der Anzahl der Bits, die im zuletzt heruntergeladenen Segment enthalten sind, zu der Zeit, die mit dem Herunterladen dieses Segments verbracht wird, in Bit pro Sekunde.
    • player_width Die aktuelle Pixelbreite des Spielers am Ende des Engagement-Bereichs.
    • player_height Die aktuelle Pixelhöhe des Spielers am Ende des Engagement-Bereichs.
    • dropped_frames: Die Anzahl der Frames, die während dieser Interaktionszeit von der Videowiedergabe verloren gegangen sind
