API-Funktionalität

Die Dynamic Ingest (DI)-API von Brightcove basiert auf einer Funktionalität, bei der Videoquelldateien vom Speicherort des Kunden heruntergeladen und bestimmte Wiedergabeversionen der Quelldateien erstellt werden. (Es besteht auch die Möglichkeit, Laden Sie Ihre Quelldateien hoch an einen temporären Speicherort, an dem Dynamic Ingest darauf zugreifen kann.) Die Plattform ist Cloud-zentriert, global verteilt und basiert auf modernen Praktiken, um die beste Konsistenz und Geschwindigkeit ihrer Klasse zu bieten.

Hinweis: Bei segmentierten Videotypen (HLS und DASH) verarbeiten einige Player den Fall nicht, in dem sich die Audio- und Videostreamlängen um mehr als die Dauer eines Segments unterscheiden. Versuchen Sie in diesem Fall eine kürzere Segmentlänge zu verwenden.

Workflow-Übersicht

Bei der gesamten Transcodierung und Speicherung von Medien werden eine Reihe von Systemen/Technologien verwendet. Sie sind:

  • Content-Management-System (CMS)-API: Erstellt ein Videoobjekt zur Verwendung in der DI API
  • Zencoder: Transkodiert das Video und erstellt mehrere Wiedergaben
  • Amazon S3: Verschiebt den Master und die Wiedergabeversionen in den Speicher, basierend auf den Profileinstellungen
  • Katalog: Speichert die mit dem Video verknüpften erforderlichen Informationen

Nach der ersten Transcodierung haben Sie die folgenden Aktionen, die Sie auf den Medien ausführen können:

  • Erneut transkodieren: Neue Darstellungen erstellen, wenn der Master vorhanden ist (Fehler, wenn der Master nicht vorhanden ist)
  • Ersetzen: Zeigen Sie auf einen neuen Master oder ersetzen Sie einen vorhandenen Master

Betrieb

Wenn Sie die DI API verwenden, führen Sie verschiedene Vorgänge aus, z. B. das Lesen eines Aufnahmeprofils und das Schreiben von Aufnahmeinformationen in Ihr Konto. Im Folgenden finden Sie eine vollständige Liste der Operationen, die für DI-Aufgaben erforderlich sind:

  • Video-Cloud/Video/Erstellen
  • Video-Cloud/Video/Lesen
  • Video-Cloud/Video/Update
  • video-cloud/ingest-profiles/profile/read
  • Video-Cloud/Ingest-Profile/Konto/Lesen
  • video-cloud/ingest-profiles/account/write
  • video-cloud/ingest-profiles/profile/write
  • Video-Cloud/Upload-URLs/lesen

Um Kundenanmeldeinformationen zu erhalten, verwenden Sie das Studio Admin-Tools oder sehen Sie sich eines der folgenden Dokumente an:

  • OAuth: Client-Anmeldeinformationen mit CURL abrufen
  • OAuth: Erhalten Sie Kundenanmeldeinformationen mit Postman

Empfohlene Vorgehensweise

  • Ein Limit von 20 Anfragen pro Sekunde für jedes Konto für die Gesamtzahl der Anfragen in DI und den CMS-API (CMS-API- oder Ingest-API-Anfragen)
  • Ein Limit von 100 gleichzeitigen DI-Jobs pro Konto (es gibt zusätzliche Optionen für große Batches von Anfragen für Konten, die in . umgewandelt wurden) Dynamische Lieferung)
  • Inhalte mit weniger als 100 GB pro Datei aufnehmen (wenn Sie größere Dateien aufnehmen müssen, wenden Sie sich bitte an Ihren Account Manager)
  • Wenn Sie Dateien in Stapeln aufnehmen, begrenzen Sie gleichzeitige Jobs mit normaler Priorität auf 100 und warten Sie, bis ein Job die Verarbeitung abgeschlossen hat, bevor Sie einen weiteren hinzufügen
  • Überprüfen Sie die Änderungen des Aufnahmeprofils, bevor Sie vollständige Batches aufnehmen
  • Du solltest nicht haben doppelte Aufträge, die für dasselbe Asset verarbeitet werden. Ein Asset ist in diesem Fall als Videoobjekt, Audiospur in einer bestimmten Sprache, Textspur in einer bestimmten Sprache oder als bestimmter Bildtyp definiert. Hier sind einige Beispiele, um den Punkt zu veranschaulichen:
    • INAKZEPTABEL: Senden Sie eine Aufnahmeanfrage für eine Videodatei und senden Sie dann eine weitere Videoaufnahmeanfrage mit derselben Video-ID, bevor der erste Aufnahmejob abgeschlossen ist
    • AKZEPTABEL: Senden Sie eine Aufnahmeanfrage für eine Videodatei, senden Sie dann eine Aufnahmeanfrage für einen Texttrack in Englisch, der derselben Video-ID zugeordnet ist, und senden Sie dann eine Aufnahmeanfrage für ein Miniaturbild mit derselben Video-ID
    • AKZEPTABEL: Senden Sie einen Ingest-Job für einen Texttrack in Englisch und senden Sie dann parallel eine Ingest-Anfrage für einen Texttrack in Spanisch
    • INAKZEPTABEL: Senden Sie eine Anfrage zum Aufnehmen eines spanischen Audiotracks und senden Sie dann sofort eine Anfrage zum Aufnehmen eines anderen spanischen Audiotracks mit derselben Video-ID
  • Die profile Feld für Aufnahmeanfragen ist Optional. Vermeiden Sie eine harte Codierung der profile in Apps und Integrationen - legen Sie stattdessen das am häufigsten verwendete Profil als das Ursprünglich für das Konto und lassen Sie das Feld weg, oder die verfügbaren Profile für das Konto abrufen und zwingen Sie den Benutzer, einen auszuwählen.

Gültige Quellstandorte

Dynamic Ingest kann Quellvideodateien abrufen von: HTTP/HTTPS oder S3 - mit oder ohne Authentifizierung

Beispiele:

  • http://example.com/path/to/input.avi
  • https://dl.dropboxusercontent.com/u/3641457/Bird_Titmouse.mp4
  • s3://my-bucket/video.mp4
Anmerkungen:
  • Videodateinamen (einschließlich der Erweiterung) dürfen 120 Einzelbyte- (60 Doppelbyte-) Zeichen nicht überschreiten. Wenn dies der Fall ist, wird das Video erfolgreich aufgenommen, aber Sie können es später nicht erneut transkodieren
  • Hostnamen (für Dateispeicherorte oder Rückruf-URLs) dürfen gemäß IETF-Spezifikationen keine Unterstriche oder bestimmte andere Satz- oder Sonderzeichen enthalten. Sehen dieser Wikipedia-Artikel für mehr Informationen. Fehler werden zurückgegeben, wenn Hostnamen nicht dem IETF-Standard entsprechen.
  • Dynamische Aufnahmedose nicht Auf Google Drive gehostete Dateien aufnehmen, da Google Drive keine direkten Links zu Videodateien generiert.

Hinweise zu S3

Wenn sich Ihre Videos in einem geschützten S3-Bucket befinden, lesen Sie Verwenden der dynamischen Aufnahme mit S3 Weitere Informationen zum Einrichten von Berechtigungen für Dynamic Ingest für den Zugriff auf Ihre Dateien.

Zu den Vorteilen der Pull-basierten Aufnahme gehören ein einfacherer Workflow und ein Repository Ihrer eigenen digitalen Master. Wenn dies jedoch keine Option für Sie ist, können Sie auch Quelldatei-Upload um Ihre Videos und andere Assets an einen temporären Speicherort hochzuladen, von dem aus Dynamic Ingest darauf zugreifen kann.

Quelldateinamen

Alle Eingabe-URLs müssen gemäß RFC 3986 beim Senden an Brightcove. Dies bedeutet, dass alle reservierte Zeichen im Pfad der URL gefunden werden, sind prozentual kodiert (Leerzeichen werden kodiert in %20), und alle reservierten Zeichen, die in der Abfrage der URL gefunden werden, werden prozentual codiert (Leerzeichen werden in codiert) + oder %20 , und + kodiert werden zu %2B).

EIN vorsigniert S3 (v2 enthält Unterschrift, Läuft ab und AWSAccessKeyId, und v4 enthält X-Amz-Algorithmus, X-Amz-Anmeldeinformationen, X-Amz-Date, X-Amz-Läuft ab, X-Amz-SignedHeaders, und X-Amz-Unterschrift) oder GCS (enthält Unterschrift, Läuft ab, und GoogleAccessId) URL sollte bereits richtig codiert sein und kann unverändert verwendet werden.

Beachten Sie, dass Doppelbyte-Unicode-Zeichen sind in Dateinamen erlaubt.

Videos aufnehmen

Zum Aufnehmen von Videos sind zwei API-Aufrufe erforderlich:

  1. Ruf die CMS-API um ein Videoobjekt im Video Cloud-System zu erstellen und seine ID abzurufen
  2. Rufen Sie die DI-API auf, um die URL für die Videoquelldatei bereitzustellen, und geben Sie die Aufnahmeprofil benutzt werden

Ein Beispielsatz grundlegender Anforderungen würde wie folgt aussehen:

CMS-API-Anfrage

HTTP-Methode
POST
Anfrage-URL
https://cms.api.brightcove.com/v1/accounts/{account_id}/videos
Anfragetext
    {
    "name": "My new video"
    }

Die Antwortdaten enthalten das Video id , die in der nächsten Anfrage verwendet wird.

API-Anfrage aufnehmen

HTTP-Methode
POST
Anfrage-URL
https://ingest.api.brightcove.com/v1/accounts/{account_id}/videos/{video_id}/aufnahmeanfragen
Anfragetext
    {
    "master": {
    "url": "http://host/master.mp4"
    },
    "profile": "high-resolution"
    }

Siehe die Schnellstart Einzelheiten zu den API-Aufrufen finden Sie auch unter a Arbeitsprobe.

Zum CMS-API Aufruf zum Erstellen des Videos im Video Cloud-System, siehe die Übersicht über die CMS-API. Beachten Sie, dass das Video name erforderlich ist und dass der Name und alle anderen Zeichenfolgen für Video-Metadaten (wie z description ) muss URI-kodiert sein.

Ersetzen eines Videos

Um ein Video durch eine neue Version oder einen neuen Satz von Wiedergaben zu ersetzen, Dynamische Aufnahme-API Anruf ist genau der gleiche wie für neue Videos aufnehmen - der einzige Unterschied besteht darin, dass Sie nicht vorher beim anrufen müssen CMS-API um das Videoobjekt im Video Cloud-System zu erstellen und eine ID dafür abzurufen. Wenn die Quellvideodatei unter der angegebenen URL dieselbe ist, die ursprünglich aufgenommen wurde, erhalten Sie einfach einen neuen Satz von Wiedergaben. Wenn die Quelldatei neu ist, ersetzen Sie das vorhandene Video. Alle Videos bleiben mit vorhandenen Wiedergabeversionen abspielbar, bis die Neutranscodierung abgeschlossen ist.

Siehe die Arbeitsprobe hier.

Ein Video neu transkodieren

Wenn du dich dafür entschieden hast einen Meister archivieren wenn Sie das Video über die aufgenommen haben Dynamische Aufnahme-API oder das Studio-Upload-Modul, dann können Sie das Video auch vom Master neu transcodieren. Auch hier ist die URL für die Aufnahmeanfrage dieselbe, aber der Anfragetext hat Folgendes:

    // request
    POST /v1/accounts/{account_id}/videos/{video_id}/ingest-requests
    // request body
    {
    "master": { "use_archived_master": true },
    "profile": "videocloud-default-v1"
    }

Anmerkungen:

  • Wenn der Master nicht archiviert wurde oder das Video nicht eingenommen mit dem Dynamische Aufnahme-API oder das Upload-Modul erhalten Sie eine Fehlermeldung
  • Die empfohlenen Standard-Aufnahmeprofile (videocloud-default-v1, screencast-1280, High Resolution, Smart-Player-Transition, Single-Bitrate-Standard und Single-Bitrate-High) werden alle den Master archivieren. Wenn Sie ein benutzerdefiniertes Aufnahmeprofil verwenden, müssen Sie angeben, dass der Meister sollte archiviert werden, wenn Sie später vom Master erneut transkodieren möchten.
  • Der Meister ist noch nie während der Neutranscodierung geändert
  • Einstellungen zum Archivieren und Verteilen von Mastern sind noch nie während der Neutranscodierung geändert
  • Bekanntes Problem: Wenn Sie ein Video mithilfe der dynamischen Aufnahme neu transcodieren, wird das Aktivierungsdatum für das Video auf das aktuelle Datum aktualisiert. Wenn Sie nach Aktivierungsdatum sortierte Smart Playlists verwenden, wirkt sich dies auf die Reihenfolge der Videos in der Playlist aus.

Bilder

Du kannst den ... benutzen Dynamische Aufnahme-API um Poster und Miniaturbilder aus Ihrem Video aufzunehmen oder eigene Bilder hinzuzufügen. Einzelheiten finden Sie unter Bilder und die Dynamic Ingest API.

Wenn Ihre Bilder aus einem geschützten S3-Bucket abgerufen werden, müssen Sie eine Bucket-Richtlinie festlegen, damit Video Cloud auf die Dateien zugreifen kann. Sehen Verwenden der dynamischen Aufnahme mit S3 für Details.

Untertitel aufnehmen

Sie können Ihrem Video auch WebVTT-Untertitel hinzufügen oder sie mithilfe von Dynamic Ingest für ein vorhandenes Video hochladen. Einzelheiten finden Sie unter Aufnahme von WebVTT-Dateien.

Anmerkungen:

  1. für Bilder und Textspuren werden nur öffentliche HTTP- oder HTTPS-URLs oder signierte S3-URLs unterstützt
  2. Für Textspuren fügt Dynamic Ingest sie nur der VTT-Liste des Videos hinzu/angehängte sie an, selbst wenn es sich um eine erneute Transcodierungsanforderung handelt. Kunde sollte die verwenden CMS-API um sie zu verwalten (aktualisieren/löschen)
  3. Dynamic Ingest akzeptiert bis zu 100 Textspuren gleichzeitig
  4. mime_type wird nicht unterstützt

DRM und HLSe

Dynamic Ingest verarbeitet Videos, die einen der von Brightcove unterstützten DRM-Typen verwenden. HLSe wird ebenfalls unterstützt.

Archivieren von Wiedergabeversionen

Standardmäßig werden alle Video- und Bildwiedergaben automatisch archiviert. Wenn Sie die Archivierung von Wiedergabeversionen deaktivieren möchten, Wenden Sie sich an den Brightcove-Support. Beachten Sie, dass digitale Meister werden archiviert, wenn dies im Aufnahmeprofil angegeben ist.

Benachrichtigungen

Sie können eine oder mehrere Callback-URLs angeben, um Benachrichtigungen über die Ergebnisse des Aufnahmeprozesses zu erhalten. Die von Ihnen angegebenen URLs sollten für Apps gelten, die POST-Anforderungen akzeptieren können. Benachrichtigungen werden im JSON-Format gesendet.

Details zum Empfang und zur Interpretation von Benachrichtigungen finden Sie in Benachrichtigungen: Dynamic Ingest- und CMS-APIs
