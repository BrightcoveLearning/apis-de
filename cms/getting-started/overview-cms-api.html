    Übersicht: CMS-API

    In diesem Thema erhalten Sie einen Überblick über die CMS-API. Die CMS API bietet ununterbrochen Lesezugriff auf die Daten. Dies ist für zeitkritische Veröffentlichungsworkflows wichtig, da Sie mithilfe von Änderungen an Videos und Wiedergabelisten vornehmen CMS API und die Daten schnell lesen, um zu überprüfen, ob sie korrekt sind.

    API-Referenz

    Siehe auch die API-Referenz.

    Allgemeine Informationen

    Basis-URL

    Die Basis-URL für den CMS API lautet:

            https://cms.api.brightcove.com/v1

    Konto-Pfad

    In allen Fällen werden Anfragen für ein bestimmtes Video Cloud Konto gestellt. Daher fügen Sie immer den Begriff accounts gefolgt von Ihrer Konto-ID zur Basis-URL hinzu:

            https://cms.api.brightcove.com/v1/accounts/{account_id}

    Authentifizierung

    Die Authentifizierung für Anfragen erfordert Authorization header :

            Authorization: Bearer {access_token}

    Der access_token ist ein temporäres OAuth2-Zugriffstoken, das vom Brightcove OAuth Dienst bezogen werden muss. Weitere Details finden Sie in der Brightcove OAuth Übersicht.

    Der einfachste Weg, Client-Anmeldeinformationen zu erstellen, sind die Verwaltungsseiten von Brightcove Studio. Ausführliche Anweisungen finden Sie unter Verwalten von API-Authentifizierungsdaten

    Betrieb

    Wenn Sie Client-Anmeldeinformationen anfordern, müssen Sie die Art des Kontozugriffs oder die gewünschten Vorgänge angeben. Im Folgenden finden Sie eine Liste der derzeit unterstützten Vorgänge für CMS API :

    • Videodaten:

      video-cloud/video/read
      video-cloud/video/create
      video-cloud/video/update
      video-cloud/video/delete
      oder
      video-cloud/video/all
      video-cloud/video/assets/delete (Wird nur benötigt, wenn Sie digitale Master löschen möchten - beachten Sie, dass Sie kann nicht Erhalten Sie diese Berechtigung, wenn Sie Anmeldeinformationen in Studio erstellen. Dies muss über die OAuth-API oder die von Brightcove Learning Services erstellte Beispiel-Apperfolgen.)

    • Playlist-Daten:

      video-cloud/playlist/read
      video-cloud/playlist/create
      video-cloud/playlist/update
      video-cloud/playlist/delete
      oder
      video-cloud/playlist/all

    • Benachrichtigungen:
      • video-cloud/notifications/all ( für Benachrichtigungen)

    Zinsbegrenzung

    Dies CMS API bietet ununterbrochen Lesezugriff auf die Daten. Dies ist für zeitkritische Veröffentlichungsworkflows wichtig, da Sie mithilfe von Änderungen an Videos und Wiedergabelisten vornehmen CMS API und die Daten schnell lesen, um zu überprüfen, ob sie korrekt sind.

    Der CMS API ist nicht für die Verwendung von Laufzeitumgebungen mit hohem Maßstab geeignet (z. B. für den Zugriff auf Videos auf einer öffentlichen Webseite mit hohem Traffic). Für Anwendungen mit hohem Datenverkehr müssen Sie eine zwischengespeicherte Schnittstelle wie: die Playback API , Galerie, Spieler oder die Native SDKs verwenden.

    Um die Leistung des Video Cloud Systems sicherzustellen, CMS API sind pro Konto nicht mehr als 20 gleichzeitige Anrufe an den zulässig. Die Zugriffshäufigkeit sollte weniger als 10 Anfragen pro Sekunde betragen.

    Gleichzeitige Anrufe treten auf, wenn mehrere parallel laufende Prozesse oder Apps API-Anfragen stellen. Da eine Anfrage mehr als eine Sekunde dauern kann, können Sie das Parallelität-Limit überschreiten, obwohl Sie nicht mehr als 10 Anfragen pro Sekunde stellen.

    Wenn sich mehrere Anwendungen in das CMS API für ein Konto integrieren, sollten diese Anwendungen Back-off- und Wiederholungslogik haben, um Instanzen zu behandeln, bei denen Parallelität oder Kurslimit erreicht werden.

    Wenn entweder das Kurs- oder das Parallelkeitslimit überschritten wird, wird ein 429 - TOO_MANY_REQUESTS Fehler zurückgegeben.

    Refert-ID-

    Um die Eindeutigkeit von Referenz-IDs zu gewährleisten, CMS API sperrt der die ID nach jeder Operation an dem Video, dem sie zugewiesen ist, bis zu 3 Minuten lang. Dies kann dazu führen, dass 409-Fehler zurückgegeben werden, wenn Sie versuchen, eine Anfrage zu wiederholen, die zu schnell fehlschlägt, oder wenn Sie versuchen, eine Referenz-ID zu früh wiederzuverwenden, nachdem Sie das zuvor zugewiesene Video gelöscht haben. Weitere Einzelheiten finden Sie in der Referenz zur Fehlermeldung .

    Limit für Videoanlagen

    Es gibt ein Limit von 1.000 Assets pro Video. Zu den Assets gehören Wiedergaben, Audiospuren, Textspuren und Bilder sowie der digitale Master. Wiedergaben und Bilder belaufen selten mehr als 10-15 Assets. Selbst wenn Sie separate Audiospuren und Textspuren für 150 verschiedene Sprachen hätten, hätten Sie immer noch weniger als 350 Assets.

    Hinweise zur Nutzung

    Der CMS API ist für Integrationen und Publishing-Workflows gedacht. Sie können diese API nicht clientseitig von einer öffentlichen Webseite aus aufrufen, da dies die Offenlegung Ihrer Client-Anmeldeinformationen für die API bedeuten würde (aus diesem Grund ist die API nicht CORS-fähig). Sie können Anrufe von einer clientseitigen (Web-) App tätigen, sofern Sie die API-Aufrufe über einen serverseitigen Proxy weiterleiten. Brightcove Learning Services bietet mehrere Beispiel-Apps , die diesen Ansatz verwenden, und einen Leitfaden zum Erstellen dieser Art von App.

    Wenn Sie alle Ihre Videos oder großen Videos abrufen möchten, sehen Sie sich die Notiz auf großen Datensätzenan.

    Vermeiden Sie hartcodierte URLs

    URLs für Thumbnails, Poster, Videodateien und andere Medien sollten auf Ihren Seiten oder Anwendungen niemals fest codiert sein. Der gibt immer CMS API die aktuellen URLs für Mediendateien zurück, aber die URLs selbst können sich ändern. Sie sollten die CMS API (oder Playback API ) -Anfragen verwenden, um diese URLs jedes Mal abzurufen, wenn die Seite geladen wird, oder sie nicht länger als sechs Stunden zwischenspeichern.

    Zwischenspeichern von Video- und Bild-URLs

    Sie können URLs für Videos und Bilder zwischenspeichern, um die Seitenleistung zu verbessern, der Cache muss jedoch regelmäßig aktualisiert werden. Wenn Sie zum Verbessern der Leistung von Seiten die abgerufenen URLs zwischenspeichern, aktualisieren Sie den Zwischenspeicher durch Wiederholen der API-Aufrufe mindestens einmal alle sechs Stunden.

    Methoden

    Derzeit unterstützt die API die folgenden Anforderungstypen:

    • GET
    • POST
    • PATCH
    • PUT
    • DELETE

    Parameter

    Bitte beachten Sie, dass alle Parameter optionalsind. Sofern nicht anders angegeben, gelten sie für GET Anfragen nach Videos und Wiedergabelisten.

    GET Anfrageparameter
    Parameter Beschreibung
    q Abfragezeichenfolge für Suchvorgänge - siehe Syntax für Suche für weitere Informationen
    limit Anzahl der zurückzugebenden Videos - muss eine ganze Zahl zwischen 1 und 100 sein. Standard: 20
    offset Anzahl der zu überspringenden Videos (für Paging-Ergebnisse). Muss eine positive Ganzzahl sein. Standard: 0
    sort Eine Zeichenfolge, die das zu sortierende Feld angibt. Beginnen Sie mit - dem Sortieren absteigend. Wenn ein Wert für angegeben q wird, ist die Standardsortierung nach „Ergebnis“ (Relevanz der Suchergebnisse für die ursprüngliche Abfrage). Wenn kein Wert für angegeben q wird, ist die Standardsortierung nach updated_at absteigend. Die folgenden Felder sind für die Sortierung gültig: name , reference_id , created_at , published_at , updated_at , schedule_starts_at , schedule_ends_at , state , plays_total , und plays_trailing_week

    Suche nach Videos

    Brightcove's CMS API bietet eine programmatische Möglichkeit, nach Videos in Ihrer Video Cloud Bibliothek zu suchen.

    Um grundlegende und komplexe Suchen Ihrer Videodaten durchzuführen, verwenden Sie den q Parameter:

            https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q={search terms}

    Einzelheiten zur Suche nach Videos finden Sie im Dokument „ Videos durchsuchen “.

    Bei Wiedergabelisten sind die unterstützten Werte für die Suchzeichenfolge begrenzter. Sie können derzeit nach suchen type , name , description , und reference_id. Hier sind einige Beispiele für gültige Suchen:

    • q=type:EXPLICIT
    • q=type:ACTIVATED_OLDEST_TO_NEWEST
    • q=type:ALPHABETICAL
    • q=bears+otters ( Name, Beschreibung oder Referenz-ID müssen entweder „Bären“ oder „Otter“ enthalten)
    • q=%2Bname:bears+type:EXPLICIT ( Name muss „Bären“ enthalten)

    Siehe Wiedergabelisten durchsuchen für weitere Details.

    Notieren Sie sich den Suchbegriff playable:true/false , mit dem Sie nur Videos zurückgeben können, die spielbar sind (oder nicht): 
            https://cms.api.brightcove.com/v1/accounts/:account_id/videos?q=%2Bplayable:true

    Diese Abfrage erfordert, dass zurückgegebene Videos abspielbar sind (nicht deaktiviert, nicht geplant und fehlen nicht abspielbare Wiedergabeversionen).

    Wenn Sie nach Videos suchen, die nicht abspielbar sind, verwenden Sie q=%2Bplayable:false.

    Paging-Ergebnisse

    Verwenden Sie den limit Parameter, um anzugeben, wie viele Elemente Sie bei einer Anfrage zurückgeben möchten - bis zu 100. Sie können den offset Parameter dann verwenden, um Ergebnismengen zu durchblättern, die größer als die sind limit. Die offset ist die Anzahl der zu überspringenden Elemente.

    Die folgende Suche gibt beispielsweise Videos 51-75 des gesamten Ergebnissatzes zurück, vorausgesetzt, die Gesamtergebnismenge enthält mindestens 75 Videos:

            /videos?q=updated_at:2014-01-01..2014-06-30&limit=25&offset=50

    Das limit und offset Parameter können sowohl für Videos als auch für Wiedergabelisten verwendet werden.

    Hinweis: limit und funktioniert offset nicht , wenn du die Videos anforderst, die zu einer Playlist gehören - diese Anfrage gibt immer alle Videos in der Wiedergabeliste zurück.

    Best Practice Paging

    Da mit der CMS-API möglicherweise gleichzeitige Änderungsvorgänge durchgeführt werden, wird empfohlen, diese Schritte auszuführen, wenn Sie Ihre Ergebnismenge durchblättern:

    1. Stellen Sie eine count Anfrage, um die Gesamtzahl der Videos in Ihrer Ergebnismenge zu erhalten. 
            /accounts/578380111111/counts/videos?q=tags:nature
    2. Verwenden Sie die limit und offset Parameter, um Datengruppen aus Ihrer Ergebnismenge zurückzugeben. 
            /accounts/578380111111/videos?q=tags:nature&limit=20&offset=50
    3. Beachten Sie, dass einige Seiten weniger als 20 Videos enthalten können. Sie werden wissen, dass Sie das Ende der Ergebnismenge erreicht haben, als Sie nach so vielen Videos gefragt haben wie in der ersten count Anfrage.

    Um es zusammenzufassen, rufen Sie die Seiten so ab, bis Sie eine Videoanzahl erhalten, die der ursprünglichen count Anfrage entspricht, da diese Zahl auf der Seite der Überschätzung übereinstimmen sollte. Fragen Sie nach:

          count / page-size + 1 page

    Sortieren von Videoergebnissen

    Verwenden Sie den Parameter sort=field_name , um anzugeben, wie die Ergebnisse sortiert werden sollen - Sie können sowohl Video- als auch Wiedergabelisten sortieren. Sie können nach folgenden Videofeldern sortieren: [1-1]

    Präfixieren Sie dem Feldnamen ein Minuszeichen ( sort= -field_name) für absteigende Reihenfolge.
    • name
    • reference_id
    • created_at
    • published_at
    • updated_at
    • schedule_starts_at ( Hinweis: dies ist das Sortierfeld - das Suchfeld ist schedule.starts_at )
    • schedule_ends_at ( Hinweis: dies ist das Sortierfeld - das Suchfeld ist schedule.ends_at )
    • state
    • plays_total [ 1 /2]
    • plays_trailing_week [ 1 /2]

    Notizen

    • [ 1-1] Wenn Sie keinen Sortierwert für einen Videosuchanruf angeben, werden die Ergebnisse nach Relevanz sortiert. Wenn Sie keinen Sortierwert für einen GET Videoaufruf angeben, werden die Ergebnisse nach updated_at absteigend sortiert.
    • [ 1-2] Sie können nach plays_total oder sortieren plays_trailing_week , aber diese Felder sind nicht in den Ergebnissen enthalten

    Sortieren von Playlist-Ergebnissen

    Sie können Wiedergabelisten nach folgenden Feldern sortieren:

    • name
    • updated_at ( Standard)
    Präfixieren Sie dem Feldnamen ein Minuszeichen ( sort= -field_name) für absteigende Reihenfolge.

    Alle Videos und große Datenmengen

    Wenn Sie alle Videos in Ihrem Konto oder eine große Anzahl von Videos abrufen, müssen Sie einige Dinge beachten:

    1. Sie könnten versucht sein, den größten zulässigen limit (100) zu verwenden, aber es ist besser, Videos in Stapeln von 25 oder weniger abzurufen, um die Möglichkeit eines Timeouts von API-Anfragen zu minimieren
    2. Da Sie große Datensätze durchlaufen, ist es möglich, dass die Videodaten während des Vorgangs aktualisiert werden, was dazu führen kann, dass sich Elemente in den Antworten verschieben:
      • Möglicherweise wird ein Artikel auf aufeinanderfolgenden Seiten wiederholt
      • Ein Artikel könnte übersehen werden, da er zu einem vorherigen Antwortset verschoben wurde

      Um der ersten Möglichkeit Rechnung zu tragen, sollte Ihre App die vollständige Artikelliste nach dem Abrufen von Videos deduplizieren. Um die zweite Möglichkeit zu nutzen, müssen Sie die Gesamtzahl der abgerufenen Elemente (nach dem De-Duping) mit der Anzahl vergleichen, die Sie erwartet haben, und dann die Anfragen erneut ausführen, indem Sie die Ergebnisse nach last_modified_date (absteigend) sortieren - Sie sollten nicht mehr als eine Charge abrufen müssen, um verpasste Artikel aufzunehmen.

    3. Sie können die Wahrscheinlichkeit der Szenarien im vorherigen Element verringern, indem Sie die zurückgegebenen Ergebnisse entsprechend sortieren. Die Standardsortierung nach Relevanz für Suchen basiert auf einem komplexen Algorithmus, der Kombinationen von Schlüsselwörtern, Tags und benutzerdefinierten Feldwerten sucht. Wenn Sie nach Videos suchen, die auf mehreren Keywords, Tags und/oder benutzerdefinierten Feldern basieren, ist die Sortierung nach Relevanz genau das, was Sie wollen. Wenn Sie jedoch nur versuchen, alle oder eine große Menge Ihrer Videos abzurufen, erhalten Sie durch explizites Festlegen des sort Parameters mehr Kontrolle über die Reihenfolge der zurückgegebenen Artikel.

    Benachrichtigungen

    Sie können Benachrichtigungen erhalten, wenn video_change Ereignisse in Ihrer Videobibliothek auftreten. Benachrichtigungen werden an die von Ihnen angegebene URL gesendet, die auf eine Anwendung verweisen sollte, die in der Lage ist, HTTP POST Anfragen zu bearbeiten.

    Hinweise:
    • Jede Änderung der Video-Metadaten wird ein video_change Ereignis auslösen. Änderungen an Assets, die vom Video verwendet werden, lösen keine Änderungsereignisse aus.
    • Zusätzlich zu Benachrichtigungen gibt es Status-APIs , mit denen Sie den Status von Aufnahmeschägen (Aufnahme, Ersetzen oder Umrechnen) erhalten, die mit einem Video verknüpft sind.

    Fehler bei Benachrichtigungen

    Das Benachrichtigungssystem behandelt jede 4xx- oder 5xx-Rückgabe vom Kundenserver als einen wiederherstellbaren Fehler. Fehlgeschlagene Rückrufe werden bis zu 20 Mal wiederholt, mit einer exponentiell steigenden Verzögerung zwischen nachfolgenden Rückrufen. Die ersten paar Wiederholungen werden innerhalb von Minuten nach dem ersten Rückrufversuch durchgeführt. Wenn der Rückruf weiterhin fehlschlägt und wir bis zur 20. Wiederholung übergehen, wird die Wiederholungsverzögerung einige Tage betragen.

    video_change -Ereignisse werden durch automatisierte Systemprozesse sowie Benutzeraktionen ausgelöst, sodass diese Ereignisse nicht immer den Änderungen entsprechen, die Sie an Videoeigenschaften in Studio oder über die APIs vorgenommen haben.

    Firewalls

    Falls Ihr Unternehmen eine strenge Richtlinie bezüglich eingehender Verkehrsquellen über Ihre Firewall hat, erlauben wir alle IPs der AWS East Region. Dies kann sich ändern, daher sollten alle AWS-IPs auf die Whitelist gesetzt werden. Weitere Informationen finden Sie unter https://docs.aws.amazon.com/general/latest/gr/aws-ip-ranges.html .
