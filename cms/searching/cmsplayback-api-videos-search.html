    CMS/Wiedergabe-API: Videos-Suche v1

    In diesem Thema erfahren Sie, wie Sie mithilfe der CMS-API nach Videos in Ihrem Konto suchen. Das CMS API bietet eine programmatische Möglichkeit, in Ihrer Video Cloud-Bibliothek nach Videos zu suchen. In diesem Thema finden Sie Details zur Suchsyntax. Hinweis: Dies ist die ursprüngliche Suchsyntax - wir empfehlen Ihnen, die einfachere Video-Such-Syntax v2zu verwenden.

    Einführung

    In diesem Thema erfahren Sie, wie Sie Folgendes tun:

    • Erstellen und begrenzen Sie eine grundlegende Suche mit dem q Parameter
    • Sortieren der Suchergebnisse
    • Suche mit den erforderlichen und ausgeschlossenen Begriffen
    • Verwenden Sie eine zitierte Suche, um exakte Begriffe und mehrere Wörter zu finden
    • Suche nach benutzerdefinierten Feldern
    • Suche nach Datumsfeldern mit einem bestimmten Datum und mit Bereichen
    • Kombination von Suchkriterien
    Beachten Sie, dass dieses Thema die Videosuche behandelt - siehe auch Playlist-Suche.

    API-Nutzung

    Die Suchfunktion kann entweder mit der Playback-API CMS API oder verwendet werden.

    Suchzeichenfolgen für eine der APIs müssen mindestens 3 Zeichen enthalten.

    CMS-API

    Wenn Sie die Suche mit der CMS-API verwenden, gilt Folgendes:

    • Alle Anfragen (einschließlich der Suche) erfordern einen Autorisierungsheader, der Ihre Zugriffstoken enthält. Weitere Informationen zum Abrufen von Client-Anmeldeinformationen und zum Abrufen von Zugriffstoken finden Sie in der Brightcove OAuth Overview.
    • Es gibt keine Begrenzung für die maximale Anzahl von Videos, die von einer Suche zurückgegeben werden, aber die Preisbegrenzung gilt.
    • Die Suchergebnisse enthalten alle Videos in Ihrem Konto, unabhängig davon, ob sie spielbar sind oder nicht, und/oder geografisch eingeschränkt.

    Details zu API-Anfragen/-Antworten finden Sie im Abschnitt „ Videos abrufen “ der CMS-API-Referenz.

    Wiedergabe-API

    Wenn Sie die Suche mit der Playback-API verwenden, gilt Folgendes:

    • Suchanfragen mit der Playback-API erfordern einen Richtlinienschlüssel, der Suchaktiviertist.
    • Es gibt ein Limit für die maximale Anzahl von Videos, die von einer Suche zurückgegeben werden.
    • In den Suchergebnissen werden nur Videos zurückgegeben, die abspielbar sind (Videos mit state:inactive werden ignoriert).
    • Suchvorgänge erzwingen Einschränkungen für die Wiedergaberichtlinie, z. B. das Auslassen von geografisch eingeschränkten Videos aus den Ergebnissen.
    • Das Zwischenspeichern der Ergebnisse bietet höhere Anforderungsraten und schnellere Antworten, und es gibt keine Ratenbegrenzung.
    Die Playback-API-Suche hat ein Limit von 100 Videos pro Anfrage mit einer Obergrenze von 1000 Videos. Erfahren Sie, wie Sie durch Suchergebnisse blättern. Wenn Sie mehr als 1000 Videos zurückgeben müssen, müssen Sie die CMS-API verwenden.

    Details zu API-Anfragen/-Antworten finden Sie im Abschnitt „ Videos abrufen “ der Referenz zur Wiedergabe-API.

    Grundlegende Suche

    Um eine Suche nach Begriffen in Ihrer Medienbibliothek durchzuführen, verwenden Sie den q Parameter.

          q={search terms}

    Die von Ihnen angegebenen Suchbegriffe sollten eine URL-kodierte Liste von Begriffen sein, die durch ein Leerzeichen getrennt sind.

    Die Suche unterstützt Stemming. Stemming ist die Zuordnung eines Wortes zur Wortwurzel und anderen Wörtern, die aus derselben Wurzel stammen. Zum Beispiel sollte eine Suche nach „Ausführen“ mit Videos übereinstimmen, die in den angegebenen Feldern „ausgeführt“ oder „ausgeführt“ sind. Es würde nicht mit „Rune“ übereinstimmen, weil „Rune“ nicht als Wurzel „gelaufen“ ist.

    Wenn Sie keinen Qualifier für einen Suchbegriff angeben, wie z. B. q=bird, sucht die Anfrage in den folgenden Feldern nach diesem Wert:

    • id [1-1]
    • name
    • description
    • long_description
    • text(kein echtes Metadatenfeld, sondern ein Pseudofeld, mit dem Sie das durchsuchen können name , description , und long_description - z.B q=text:bird)
    • tags
    • reference_id
    • custom_fields ( durchsucht alle benutzerdefinierten Felder)
    • custom_field_name ( durchsucht ein bestimmtes benanntes benutzerdefiniertes Feld)[1-2]

    Notizen

    [ 1-1] Hinweis: Die Suche nach ID ist aus Konsistenzgründen enthalten, aber eine Suche nach q=id:12345 liefert genau die gleichen Ergebnisse wie die Anfrage https://cms.api.brightcove.com/v1/accounts/{account_id}/videos/12345
    [ 1-2] Wenn Sie ein benutzerdefiniertes Listenfeld haben und Videos mit einem von mehreren Werten zurückgeben möchten, können Sie dies folgendermaßen tun:

    Nehmen wir an, Sie haben ein Feld namens color , das die Werte annehmen kann: red , green , yellow , oder blue. Sie möchten Videos finden, bei denen dieses Feld auf den Wert festgelegt ist green oder blue:

          q=color:green%20color:blue

    Beispiel: Diese Anfrage gibt Videos zurück, deren Wert bird in mindestens einem der oben aufgeführten Felder enthalten ist.

          https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q=bird
    Suchzeichenfolgen mit weniger als 3 Zeichen funktionieren nicht - stattdessen werden alle Videos zurückgegeben.

    Suche auf eine bestimmte Eigenschaft beschränken

    Wenn Sie einen Qualifizierer für einen Suchbegriff angeben, wie z. B. q=name:bird, durchsucht die Anfrage das name Videofeld nach einem Wert von bird.

    Beispiel: Diese Anfrage gibt Videos zurück, deren Wert wildlife im name Feld enthalten ist.

          https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q=name:wildlife

    Die unterstützten Suchfelder sind:

    Unterstützte Suchfelder
    Feld Gesetzliche Werte
    name Zeichenfolgen oder Zeichenfolgen in Anführungszeichen
    Text Zeichenfolgen oder Zeichenfolgen in Anführungszeichen (durchsucht die name , description , und long_description)
    tags Strings oder zitierte Strings (mehrere Tags sollten durch Kommas getrennt sein)
    custom_fields Strings oder zitierte Zeichenfolgen (durchsucht alle benutzerdefinierten Felder - Sie können auch einen internen Namen eines bestimmten benutzerdefinierten Feldes verwenden) [2-1]
    reference_id string oder zitierte Zeichenfolge
    state ACTIVE, INACTIVE, PENDING, DELETED [2-3]
    updated_at Datumsbereich
    created_at Datumsbereich
    schedule.starts_at Datumsbereich
    schedule.ends_at Datumsbereich
    published_at Datumsbereich
    complete [ 2-2] true oder false

    Hinweise

    • [ 2-1] Es ist nicht möglich, nach Videos zu suchen, die ein benutzerdefiniertes Feld haben null, das keinen Wert oder einen Wert von hat, da das Feld nicht einen Wert erhalten hat in den Video-Metadaten überhaupt enthalten.
    • [ 2-2] Wenn Sie ein neues Video erstellen, wird die complete Eigenschaft automatisch auf eingestellt false. Sobald eine Wiedergabeversion für das Video existiert, wird die complete Eigenschaft automatisch auf festgelegt true.
    • [ 2-3] Die folgenden Einschränkungen gelten für die Suche nach GELÖSCHTEN Videos:
      • Die Suche nach gelöschten Videos ist nur über die CMS-API verfügbar. Die Playback-API gibt keine gelöschten Videos zurück
      • Nur Videos, die in den letzten 10 Tagen gelöscht wurden (die aktuelle Zeit minus 10 Tage), werden zurückgegeben
    Die Felder, die einen Zeichenfolgenwert verwenden, führen eine Textsuche für dieses Feld durch.

    Sortierung von Suchergebnissen

    Der sort Parameter ermöglicht es Ihnen, Ergebnisse einer Get Request für Videos zu sortieren. Sie können nach folgendem sortieren:

    • reference_id
    • name
    • created_at
    • published_at
    • updated_at
    • schedule.starts_at
    • schedule.ends_at
    • state
    • plays_total
    • plays_trailing_week

    Wenn die Ergebnisse nicht explizit durch die Verwendung von sortiert werden sort, werden die Ergebnisse nach einem Algorithmus sortiert, der als Term Frequency/Inverse Document Frequency oder bekannt ist TF-IDF. Weitere Informationen finden Sie hier .

    Angenommen, Sie suchen nach Begriffen coastal,city In Ihrem Konto befinden sich 120 Videos, deren Begriffe irgendwo in den Videometadaten enthalten sind ( name , description , tags und so weiter) und das entspricht auch den Sortierkriterien für die Ergebnisse (zum Beispiel haben sie alle das gleiche schedule_starts_at Terminzeit). Wie hoch in den Ergebnissen ein Video erscheint, hängt von der Häufigkeit ab, mit der ein oder beide Terme in den Metadaten erscheinen, wobei der Begriff, der am häufigsten in Ihrer Videobibliothek als Ganzes erscheint, ein größeres Gewicht hat.

    Erforderliche/ausgeschlossene Bedingungen

    Sie können einen Suchbegriff als erforderlich markieren (zurückgesendete Videos MÜSSEN übereinstimmen) oder ausgeschlossen (zurückgegebene Videos dürfen NICHT übereinstimmen). Dies wird mit einem URI-codierten + (%2B) oder - Zeichen kontrolliert, das dem Begriff unmittelbar vorausgeht.

    Sie müssen + wie %2B bei der Verwendung codieren, um einen erforderlichen Begriff anzugeben.

    Bedarfliche/ausgeschlossene
    Beispiel URL-kodiert Sinn
    +foo %2Bfoo Videos MÜSSEN den Begriff foo in der name , description , long_description , tags , reference_id oder custom_fields
    +custom_fields:foo %2Bcustom_fields:foo Video MUSS den Wert foo für ein benutzerdefiniertes Feld enthalten
    +foo -bar %2Bfoo%20-bar Videos MÜSSEN den Begriff enthalten foo darf aber NICHT den Begriff enthalten bar in dem name , description , long_description , tags , reference_id oder custom_fields
    +name:foo -name:bar %2Bname:foo%20-name:bar Videos MÜSSEN den Begriff enthalten foo , dürfen aber NICHT den Begriff bar in der name

    Beispiel: Diese Anfrage gibt Videos zurück, die einen Wert von haben, sea aber KEINEN Wert von lake im tags Feld haben.

          https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q=%2Btags:sea%20-tags:lake

    Siehe Kombinieren von Suchkriterien unten, um zu erfahren, wie Sie die erforderliche/ausgeschlossene Syntax verwenden, um UND-Logik für mehrere Suchbegriffe durchzusetzen.

    In Kombination mit anderen Params

    Suche (mit dem q Parameter) kann mit anderen Parametern wie z sort , limit und offset. Alle URL-Parameter sind durch getrennt &. Die Reihenfolge der Parameter spielt keine Rolle.

    Search param limit hat einen maximalen Wert von 100 pro Anfrage. Bitte beachten Sie die Ergebnisse der Seitensuche bezüglich der Videopagierung.

    Beispiele

    Beispiel: Diese Anfrage gibt Videos zurück, die einen Wert von bar im tag Feld haben müssen und möglicherweise einen name enthaltenden Wert haben foo

          .../videos?q=name:foo%20%2Btags:bar&sort=updated_at

    Beispiel: Diese Anfrage gibt dieselben Videos wie oben zurück, sortiert diese Ergebnisse jedoch zusätzlich nach dem Feld updated_at und beschränkt die Ergebnisse dann auf nur 10 Videos.

          .../videos?sort=updated_at&q=name:foo%20%2Btags:bar&limit=10

    Zitierte Suchbegriffe

    Standardmäßig stimmt eine Suche mit ähnlichen Wörtern mit Ihren Suchbegriffen überein. Wenn Sie mehrere Wörter finden möchten, wickeln Sie den Begriff einfach in Anführungszeichen ein.

    Die meisten Browser und andere Agenten behandeln wörtliche Anführungszeichen ("...") korrekt, aber wenn Sie auf einen Fall stoßen, in dem zitierte Suchbegriffe scheinbar keine korrekten Ergebnisse liefern, ersetzen Sie die Anführungszeichen durch %22 (%22...%22)

                
              q="foo" or q=%22foo%22
              q="foo%20bar" or q=%22foo%20bar%22
    Beachten Sie, dass für die Anführungszeichen übereinstimmende Videos erforderlich sind, die alle Begriffe enthalten, jedoch nicht unbedingt exakte Übereinstimmungen zu diesen Bedingungen. Beispielsweise "home run" könnte eine Suche nach ein Video mit den Wörtern homeless und running im Titel zurückgeben.

    Dies funktioniert auch beim Suchen nach einem bestimmten Feld:

                
              q=name:"home"
              q=name:"home%20run"

    Mehrere Worte

    Beispiel: Beachten Sie, dass diese Anfrage Videos zurückgibt, die entweder einen Wert sea oder mammal im tags Feld haben.

          https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q=tags:sea,mammal

    Die folgende Anfrage gibt jedoch nur die Videos zurück, die ein Tag haben sea,mammal.

          https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q=tags:"sea,mammal"
    Tags sind ein Sonderfall, in dem Sie nach mehreren Tags als kommagetrennte Liste suchen können - das gilt nicht für andere Felder wie Name, Beschreibung usw. Kombinieren Sie für andere Arten von Mehrfachbegriffen Begriffe mit + Zeichen wie folgt: q=tags:"bird"%20name:"owl".

    Benutzerdefinierte Felder

    Sie können nach jedem benutzerdefinierten Feld suchen, das Sie für Ihre Videos definiert haben.

          q=my_field:foo
      q=my_field:"foo"

    Hinweis: Alle benutzerdefinierten Feldwerte werden als Strings behandelt. Zum Beispiel, wenn Sie ein benutzerdefiniertes Feld vom Listentyp haben, das die Werte annehmen kann true oder false Bei der Suche wird nach diesen Zeichenfolgen gesucht, nicht nach booleschen Werten (in vielen Programmiersprachen). 1 und 0 kann austauschbar mit verwendet werden true und false als boolesche Werte, aber weiter suchen q=my_boolean_field:1 gibt keine Videos zurück, die haben my_boolean_field einstellen true).

    Beispiel: Diese Anfrage gibt Videos zurück, die einen Wert von Tier im subject benutzerdefinierten Feld haben.

          https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q=subject:animal

    Datum-Bereiche

    Wenn Sie in einem Datumsfeld suchen, können Sie ein bestimmtes Datum oder einen Datumsbereich angeben, indem Sie zwei Perioden verwenden, um das Start- und Enddatum zu trennen (q=updated_at:2018-01-01..2018-02-01).

    Dadurch wird nach allen Videos mit einem updated_at Wert zwischen dem 1. August 2012 und dem 8. Oktober 2012 gesucht. Hier geben wir jedes Datum im UTC-Format an.

          q=updated_at:2012-08-01T00:00:00Z..2012-10-08T23:59:59Z

    Sie können diese Suche vereinfachen, indem Sie die Zeitkomponenten löschen. Folgendes entspricht der obigen Suche.

          q=updated_at:2012-08-01..2012-10-08

    Unterstützte Datumsform

    Die unterstützten Datumsformate umfassen UTC und relative.

    Beispiele für Datumsangaben
    format (URI-kodiertes Format) Sinn
    2015-08-01T 06:15:00 Z Dies entspricht einer Zeit in UTC.
    2012-08-01 Dies entspricht Mitternacht an einem Tag in UTC. Das Beispiel entspricht 2012-08-01T 00:00:00 Z
    -1d Die aktuelle Zeit minus 1 Tag. (siehe unten)

    Relative Datumsangaben

    Für relative Termine unterstützen wir eine Richtung ( + oder - ) gefolgt von einer Zahl, gefolgt von einer Dauer. Relative Daten werden immer ab der aktuellen Zeit gemessen. Die gesetzliche Dauer ist: Minuten, Stunden, Tage.

    Die + Zeichen müssen uri-codiert sein als %2B.

    Beispiele:

    Beispiele für relatives Datum
    q param für Datumsangaben Bedeutung
    q=updated_at ፦1day.. JETZT Videos aktualisiert von vor 1 Tag auf den aktuellen Tag
    q=created_at ፦2Tage videos Hinzugefügt vor 2 Tagen
    q=updated_at ፦4hours.. JETZT Video aktualisiert von vor 4 Stunden auf die aktuelle Zeit
    q=created_at ፦60 Minuten.. Videos wurden von vor 60 Minuten zur aktuellen Zeit hinzugefügt
    q=created_at:2016-01-01.. -1d Videos erstellt vom 1. Januar 2015 bis vor einem Tag
    q=updated_at ፦14d.. JETZT Videos der letzten zwei Wochen

    Offene Bereiche

    Lassen Sie den Anfang oder das Ende des Bereichs weg, wenn alle Daten bis zu einem bestimmten Zeitpunkt oder alle Daten ab einem bestimmten Zeitpunkt berücksichtigt werden sollen.

    Beispiel: Suche nach allen Videos, die in den letzten 2 Tagen geändert wurden

          q=updated_at:-2days..

    Beispiel: Suche nach allen Videos, die am oder nach dem 11. August 2013 geändert wurden:

          q=updated_at:2013-08-11T00:00:00Z..

    NOW Betreiber für Zeitermine

    Für schedule.starts_at und schedule.ends_at können Sie NOW als Datumswert verwenden. Dies ist ein Komfort-Operator, mit dem Sie eine dynamische Abfrage basierend auf dem aktuellen Datum und der Uhrzeit einrichten können. Ein paar Beispiele:

    Beispiele für Zeitplandaten
    von/nach params URI-codiert Bedeutung
    ?q=schedule.starts_at:.. JETZT ?q=schedule.starts_at:.. JETZT starts_at ist vom Anfang der Zeit bis zu diesem Moment
    ?q=schedule.starts_at:jetzt ?q=schedule.starts_at:jetzt starts_at ist von diesem Moment an
    ?q=schedule.ends_at:Jetzt.. ?q=schedule.ends_at:Jetzt.. ends_at ist von diesem Moment bis zum Ende der Zeit
    ?q=+schedule.starts_at:.. JETZT +schedule.ends_at:jetzt.. ?q=%2bSchedule.starts_at:.. NOW% 20% 2bSchedule.ends_AT:Now.. starts_at vor diesem Moment und ends_at nach diesem Moment (Video ist in diesem Moment im Zeitplan)

    Kombination von Suchkriterien

    Sie können Kriterien für komplexe Suchen kombinieren.

    Beispiel: Diese Anfrage sucht nach Videos mit einem name Wert von Klatsch , die zwischen dem 1. August 2010 und dem 8. Oktober 2010 aktualisiert wurden. Anschließend werden die Antwortdaten in absteigender Reihenfolge nach updated_at Datum sortiert.

          q=%2Bname:gossip%20%2Bupdated_at:2010-08-01..2010-10-08&sort=-updated_at

    Begriffe kombinieren

    Verwenden Sie die erforderliche/ausgeschlossene Syntax , um Videos zurückzugeben, die alle angegebenen Begriffe enthalten.

    Beachten Sie, dass in allen Beispielen mehrerer Suchbegriffe, die folgen, dass Textsuchbegriffe nach zusätzlichen Begriffen nach dem ersten Term ein URIS-codiertes Leerzeichen (%20) haben, so dass der zusätzliche Begriff nicht als Teil des ersten interpretiert wird.

    Zum Beispiel, wenn Sie nach suchen:

          q=name:foo +tags:bar (URI-encoded: q=name:foo%20%2Btags:bar)

    Die Antwort enthält Videos, die das Tag „Leiste“ haben und möglicherweise auch foo im Namen enthalten sind. Wenn Sie nur die Videos zurückgeben möchten, die den Namen UND den Tag „Leiste“ enthalten, müssen Sie nach folgender Suche suchen: foo 

          (unencoded) q=+name:foo +tags:bar (URI-encoded) q=%2Bname:foo%20%2Btags:bar

    Wenn Sie nur Videos zurückgeben möchten, die foo im Namen enthalten sind, aber nicht über das Tag „Leiste“ verfügen, suchen Sie nach:

          (unencoded) q=+name:foo -tags:bar (encoded) q=%2Bname:foo%20-tags:bar

    Beispiele

    Beispiele: Combining Be
    Uncodierte Suchzeichenfolge URI codierter Suchstring Bedeutung
    q=foo bar q=foo%20bar zurückgegebene Artikel haben „foo“ ODER „bar“
    q=foo +bar q=foo%20%2Bbar zurückgegebene Artikel müssen „bar“ haben, kann „foo“ haben
    q=+foo bar q =%2Bfoo%20bar zurückgegebene Artikel müssen „foo“ haben, möglicherweise „bar“
    q=+foo +bar q=%2Bfoo%20%2Bbar zurückgegebener Artikel muss „foo“ UND „bar“ haben
    q=-foo +bar q=-foo%20%2Bbar Der zurückgegebene Artikel muss „bar“ haben UND darf nicht „foo“ haben
    Mehrere Tag-Suchen
    q=tags:bee,bop q=tags:bee,bop gibt Videos mit dem Tag „bee“ ODER „bop“ zurück
    q=tags:bee tags:bop q=tags:bee%20tags:bop gibt Videos mit dem Tag „bee“ ODER „bop“ zurück
    q=+tags:bee tags:bop q=%2Btags:bee%20tags:bop Alle zurückgegebenen Videos müssen das Tag „bee“ haben. Möglicherweise haben sie auch das Tag „bop“
    q=+tags:bee +tags:bop q=%2Btags:bee%20%2Btags:bop alle zurückgegebenen Videos haben das Tag „bee“ UND das Tag „bop“

    Suche nach bestimmten Videos

    Wenn Sie Ihre Suche auf eine bestimmte Gruppe von Videos beschränken möchten, können Sie Folgendes tun id:

    Beispiel: Diese Anfrage gibt Videos mit IDs 123456789 zurück 987654321 und 48376387

          q=id:123456789%20id:987654321%20id:48376387

    Suche nach Bundesland

    Sie können Suchen nach dem Status des Videos mithilfe des Parameters ausführen oder filtern:

          q=state:ACTIVE( | INACTIVE | PENDING | DELETED)[3]

    Hinweise

    • [ 3] Die Suche nach GELÖSCHTEN VIDEOS ist nur für Videos verfügbar, die in den letzten 10 Tagen (die aktuelle Zeit minus 10 Tage) und nur über die CMS API(nicht die Playback-API) gelöscht wurden.

    Stemming

    Stemming wird unterstützt, aber keine partielle Wortsuche. Zum Beispiel, q=name:ban sollte Videos mit den Namen zurückgeben "Parking Ban Announced " oder "Parking to be Banned " oder "City Banning Parking " aber nicht "Bank Holiday " oder "Bandit Captured".

    Leerzeichen/Sonderzeichen

    Im CMS API Allgemeinen verarbeitet der Sonderzeichen in Suchzeichenfolgen mit ein paar Ausnahmen:

    • Leerzeichen sind nicht erlaubt und müssen als kodiert sein %20. (Nicht codierte + Zeichen können auch Leerzeichen ersetzen, dies kann jedoch zu Verwirrung bei Ihren Abfragen führen, da + dies auch darauf hinweist, dass ein Begriff erforderlich ist. Siehe erforderliche/ausgeschlossene Syntax)

      Um zum Beispiel nach „meinem Lieblingsvideo“ im Namen zu suchen:

      q=name:"my%20favorite%20video"

      Beachten Sie, dass die Anführungszeichen sicherstellen, dass zurückgegebene Videos alle Bedingungen enthalten. Ohne sie würden Videos mit einem der drei Begriffe zurückgegeben. Mehr dazu weiter unten.
    • Um nach einem wörtlichen + Zeichen zu suchen oder den zu verwenden, + um anzugeben, dass die zurückgegebenen Videos einen Begriff enthalten müssen , müssen Sie den + als %2B:

      Suche nach Videos, die "two+two" im Namensfeld enthalten sein müssen

      q=name:two%2Btwo

      Suche nach Videos, die "heron" im Namensfeld enthalten sein müssen

      q=%2Bname:heron

    • Einige Agenten behandeln möglicherweise nicht korrekt mit wörtlichen Anführungszeichen, daher ist es sicherer, "foo" als %22foo%22
    Hinweise:

    Bei Mehrwort-Tags oder Strings schließen Sie das Tag/die Zeichenfolge in uri-codierte Anführungszeichen ( %22) ein und codieren Sie die Leerzeichen mit URI (%20):

    Multi-Wort-Tag-Beispiel
    Nicht codierter Suchstring Codierter Suchstring
    q=tags:"multi word tag" q=tags:%22multi%20word%20tag%22

    Die Anführungszeichen stellen sicher, dass zurückgegebene Videos MIT ALLEN Bedingungen übereinstimmen, jedoch NICHT unbedingt mit den genauen Bedingungen. Stemming gilt weiterhin - zum Beispiel "new house" wird eine Suche nach auch Videos mit einem name oder description ähnlichen zurückgeben news about housing.

    Für einmalige Anfragen können Sie den String-Encoder von Brightcove Learning verwenden, um Ihre Suchzeichenfolgen zu kodieren . Für Apps müssen Sie eine URI-Codierungsfunktion in der von Ihnen verwendeten Sprache finden.

    Sternchen ( *) werden als gewöhnliche Sonderzeichen behandelt, nicht als Platzhalter.

    Clip Suchbegriffe

    Clips sind Videos, die aus Abschnitten anderer Videos erstellt wurden. Clips können von Brightcove Social generiert werden, und andere Mittel werden in Zukunft verfügbar sein. Es gibt einige spezielle Suchbegriffe, die Sie verwenden können, um generierte Clips in einem Konto zu finden:

    • q=%2Bis_clip:true - gibt nur Clips zurück
    • q=%2Bis_clip:false - gibt nur Nicht-Clips zurück
    • q=%2Bclip_source_video_id:video_id - gibt Clips zurück, die aus dem angegebenen Video generiert wurden

    Ignorierte Worte

    Bestimmte Wörter werden in Suchzeichenfolgen ignoriert, da sie so häufig sind, dass sie wahrscheinlich viele Ergebnisse zurückgeben, die nichts mit dem zusammenhängen, wonach Sie tatsächlich suchen. Im Folgenden finden Sie eine Liste von Wörtern, die bei der Suche ignoriert werden:

    „a“, „an“, „und“, „sind“, „als“, „bei“, „bei“, „aber“, „bei“, „wenn“, „in“, „in“, „in“, „ist“, „es“, „nein“, „nicht“, „von“, „an“, „oder“, „so“, „ihr“, „dann“, „dort“, „diese“, „sie“, „dies“, „zu“, „was“, „werden“, „mit“

    Bekannte Probleme

    • Doppelte Ergebnisse: In bestimmten Fällen werden einige Elemente in den Suchergebnissen möglicherweise mehr als einmal angezeigt.

      Problemumgehung: Um doppelte Suchergebnisse zu verhindern, verwenden Sie immer einen sort Parameter in Ihren Suchanfragen.
